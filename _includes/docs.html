Blinking a light is the "Hello World!" of programming devices. So let's by design a device API by taking simple steps between an API server and an API client and work our way toward a fully compliant Zetta API for a light bulb connected to the Web.

Properties

State

Style

Custom

We'll host our light at http://hello.zettaapi.org with a URL design that can handle our light and future devices of various types. We'll also account for intermediate hardware gateways like homes automation hubs. Let's use /servers/{id}/devices/{id} and make a GET request to it:

GET http://hello.zettaapi.org/servers/1/devices/1

Actions

Eventually the server will provide a lot of information about our devices. To start, though, how should the server respond to indicate the light is off? We'll design a properties section in our response to hold the state property:

{
  "properties": {
    "state": "off"
  }
}

Links

WebSockets

Now the client knows that the light is off. But how would the client turn it on? The client developers could the read API docs. Or better, let's have the server tell the client within the API response by including an actions section:

{
  "links": [
    { "title": "logs",
      "href": "ws://hello.zettaapi.org/servers/1/events?topic=light%2F1%2Flogs"
    }
  ]
}

Entities

Now that the client understands how to turn on the light, let's have the client POST to the light's URL with action=turn-on:

POST http://demo.zettaapi.org/servers/1/devices/1
action=turn-on

The server will turn the light on and then respond with the new current state. But what about actions? Now that the light is on, the server needs to tell the client how to turn it off:

{
  "properties": {
    "state": "on"
  },
  "actions": [{
    "name": "turn-off",
    "method": "POST",
    "href": "http://hello.zettaapi.org/servers/1/devices/1",
    "fields": [{
      "name": "action",
      "value": "turn-off"
    }]
  }]
}
